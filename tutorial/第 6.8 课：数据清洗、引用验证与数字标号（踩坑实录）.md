## 一、数据清洗：EPUB 解析与去噪

### 1.1 踩坑：EPUB 章节标题识别失败

**现象**：入库《隋唐制度渊源略论稿》后，全部 489 个 chunk 的 `parent_text` 都是 `# 开篇`，章节路径完全丢失。

**根因**：该书使用 `<p><font size="6"><b>礼仪</b></font></p>` 作为章节标题，而非标准 `<h1/h2>` 或 `<p>第X章</p>`。`epub_parser.py` 的 `_convert_element()` 只处理：

```python
# 原代码（缺陷版）
elif tag_name == 'p':
    text = ' '.join(content.split())
    if re.match(r'^第[一二三四五六七八九十百零\d]+章\s*$', text):
        return f"\n# {text}\n"
    return f"{text}\n\n" if text else ""
```

`<p><b>礼仪</b></p>` 被当成普通段落，没有生成 `# 礼仪` 标题。

**修复方向**（预留扩展）：
```python
# 检测 <p> 下的 <b>/<strong> 子标签，且字体较大（如 font size >= 5）
# 或文本长度较短（<20字）+ 居中属性，作为 H1/H2 候选
```

> **教训**：EPUB 没有统一标准，同一套解析器面对不同出版社的电子书会翻车。入库前务必先抽样检查 HTML 结构。

### 1.2 踩坑：去噪正则误杀 Markdown Heading

**现象**：入库《剑桥中国隋唐史》后，章节标题变成 `# —617年`，前面内容被截断。

**根因**：`TextCleaner.denoise()` 中有一条正则本意是去除"混入正文的章节标题+页码"：

```python
# 致命 bug：过于贪婪
text = re.sub(r"第[一二三四五六七八九十百]+章.*?\d+", "", text)
```

对于 `# 第二章 隋朝，581—617年`：
- `第二章` → 匹配 `第[数字]+章`
- `.*?` → 匹配 ` 隋朝，`
- `\d+` → 匹配 `581`
- 结果：`第二章 隋朝，581` 被整个删除，只剩 `# —617年`

**修复**：
```python
# 要求数字前无空格/标点，避免误杀正常 heading
text = re.sub(
    r"第[一二三四五六七八九十百]+章[^\s\d,，.。;；:：\-\—]{2,}\d+",
    "", text
)
```

> **教训**：文本清洗的正则要"保守"，宁可漏杀不要误杀。特别是 EPUB 解析后的 Markdown 文本，heading 行是结构命脉。

## 三、引用验证（Citation Verifier）

### 3.1 问题背景：Citation Hallucination

LLM 知道正确答案（来自参数记忆），但检索器没召回支撑片段。模型硬把 `[1][2]` 贴到无关 chunk 上，假装有据可查。

### 3.2 实现：规则硬匹配（约束条件下的最优解）

**调研过但放弃的路线**：
- 语义相似度（embedding cosine）：对古文效果差，"以人为镜"和"可以明得失"语义相关但不在同一 chunk
- LLM Judge：成本高、延迟大
- NLI（自然语言推断）：需要微调，数据难搞

**最终选择**：jieba POS 提取"硬事实" + 关键词匹配

```python
def _extract_key_terms(sentence: str) -> List[str]:
    # 1. 提取直接引语（加引号的内容，不可转述）
    # 2. 提取数字（年份、税率、数量）
    # 3. jieba 分词：保留名词(n)、人名(nr)、地名(ns)、专有名词(nz)
    # 4. 过滤通用干扰词（如"唐太宗""贞观"在所有 chunk 中都出现）
```

### 3.3 踩坑清单

| 坑 | 现象 | 修复 |
|---|---|---|
| **空字符串** | `core_total=0` 时 `core_matched/core_total` 导致除零，fallback 到 `len(matched)/len(core_terms)` 后空字符串 `''` 永远 `in chunk_text` | `_is_core_entity` 要求 `len(term) >= 3` |
| **中文引号** | `re.findall(r'"(.+?)"', sentence)` 匹配不到中文引号 `"..."` | 分别处理四种引号：英文双引、中文双引、英文单引、中文单引 |
| **jieba POS 古文标注错误** | `"以人为镜"` 标成 `l`（习用语），`"得失"` 标成 `v`（动词） | 引语内容不过滤词性，直接保留 |
| **引用标记干扰分词** | `[1]` 被 jieba 切成 `[、1、]`，混入 key_terms | 预先用正则 `re.sub(r'\[\d+\]', '', sentence)` 去掉引用标记 |
| **动态通用词** | 尝试从召回 chunks 交集中自动计算 `generic_terms`，jieba 对古文 POS 标注不稳定导致大量 false positive | **回滚到硬编码**：`{'贞观', '唐太宗', '太宗', '唐初', '唐朝', '唐代', ...}` |

> **面试叙事要点**：三层演进——调研主流方案（语义/LLM Judge/NLI）→ 约束条件下选择规则硬匹配 → 尝试动态通用词 → 评估后回滚到硬编码。**展现工程权衡思维**。

### 3.4 动态阈值

```python
def _resolve_threshold(answer: str, base_threshold: float) -> float:
    # 关系/推断类问题放宽，事实类问题收紧
    relation_keywords = ['关系', '影响', '作用', '联系', '区别', '比较', '意义']
    if any(w in answer for w in relation_keywords):
        return max(0.15, base_threshold - 0.1)  # 关系类 0.2
    return base_threshold  # 事实类 0.3
```

## 四、数字标号：引用编号重映射

### 4.1 问题：LLM 引用编号不连续

LLM 实际引用了 source_map 中的 `[3]` 和 `[5]`，用户看到的不连续编号很突兀。

### 4.2 实现：按出现顺序重映射为连续编号

```python
def _format_citation_footer(self, answer: str, source_map: SourceMap):
    # 1. 提取所有引用标记，保持出现顺序
    cited_ids_ordered = []
    for match in re.finditer(r'\[(\d+)(?:·存疑)?\]', answer):
        cid = match.group(1)
        if cid not in seen and cid in source_map:
            cited_ids_ordered.append(cid)

    # 2. 创建重映射表：原编号 → 新连续编号
    id_remap = {old_id: str(new_idx) for new_idx, old_id in enumerate(cited_ids_ordered, 1)}

    # 3. 替换 answer 中的编号
    for old_id, new_id in id_remap.items():
        answer = answer.replace(f"[{old_id}]", f"[{new_id}]")

    # 4. 底部追加标准化引用列表
    lines = ["\n---\n**引用来源：**"]
    for old_id in cited_ids_ordered:
        new_id = id_remap[old_id]
        source = source_map[old_id]
        lines.append(f"[{new_id}] 《{doc_short}》· {chapter_title}")

    return answer, "\n".join(lines), id_remap
```

> 结果：`[3][5]` → 用户看到 `[1][2]`，底部列出 `[1] 《唐太宗传》· 第一节 ...` 和 `[2] 《唐太宗传》· 第三节 ...`

### 4.3 踩坑：CRITICAL 引用直接删除 → "裸奔事实"

**早期做法**：发现 CRITICAL 虚假引用后直接删除 `[1]`，导致句子变成"这种因果关系是明确的。"——有事实声明但无引用支撑。

**修复**：降级为 `"（史料记载）"`，避免裸奔：
```python
if repaired:
    answer = answer.replace(f"[{cid}]", f"[{repaired}]")
else:
    answer = answer.replace(f"[{cid}]", "（史料记载）")
```

### 4.4 清理 LLM 自发生成的引用块

LLM 有时在回答末尾自己生成 `---
**引用来源：**...`，与我们的 footer 重复。

```python
def _strip_llm_citation_footer(answer: str) -> str:
    patterns = [
        r'\n---+\s*\n?\*\*引用来源：.*?$',
        r'\n\*\*引用来源：.*?$',
        r'\n来源[：:].*?$',
        r'引用来源[：:].*?$',
        r'\n参考文献[：:].*?$',
    ]
    for pat in patterns:
        answer = re.sub(pat, '', answer, flags=re.DOTALL)
    return answer.strip()
```
